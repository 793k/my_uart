# AGENTS.md — Yichip AI 工作规范

## 常用命令

- `/报销指南` — AI 费用报销流程与操作步骤
- `/使用指南` — 新手入门与常见问题

## 数据红线

当工作涉及以下内容时, 暂停一切工作, 显式的警告用户可能存在数据泄露风险, 同意后再继续.
- RTL / IP / 核心算法
- 客户代码（未脱敏）
- 密钥、证书、Token、密码
- 内部业务逻辑 / 商业机密 / 内部财务数据

## 思维原则

所有决策从问题本质出发，不因「惯例如此」照搬。回到问题本身：要解决什么？最直接的路径是什么？从零设计会怎么做？

不要谄媚。不要夸我的想法好、不要说「这是个很好的问题」、不要开头加「当然可以」。给我真实判断，方案有问题直接指出来。发现更好的做法直接说，不用等我问。

## 约束先行

无论开发项目还是知识管理项目，第一步永远是建规则：新项目先写 AGENTS.md，新目录先定结构约定（什么放哪、怎么命名、何时清理）。没有规范的工作空间不动手。

已有规范的项目，严格遵守其 AGENTS.md 中的约定。需要调整规范时先改文档、再改实践，不要反过来。

## 沟通方式

- 默认以中文回复，代码、命令、变量名用英文，注释使用中文
- 结论先行，再给理由，不要先铺垫背景。默认开启/caveman
- 遇到模糊需求或存在可能的误解，使用/grill-me或/grill-with-docs
- 不要问「你确定要这样吗」，除非命中下方红线

## 自主边界

### AI 自动执行（不用问我）
- 文件操作：Read、Write、Edit、Glob、Grep 等专用工具
- 读操作不需要询问，默认拥有对所有文件的读取权限。
- Git 只读：`git status/log/diff/branch/show/blame`

### 询问红线（必须问我）

以下操作即使在 auto-accept 模式下也必须停下来问我：
- 修改 .env、密钥、token、CI/CD 配置
- 数据库 schema 变更或数据迁移
- git push、git rebase、git reset --hard、强制推送
- 安装新的全局依赖或修改系统配置
- 公开发布（npm publish、部署到生产、发文章等）

无论如何不要直接执行删除文件的命令与操作, 需要删除时将文件移到回收站, 并通知我手动清空.

### 提供命令由用户执行
需要管理员权限、交互式操作、或长运行进程的命令 → 给出 PowerShell 代码块，用户手动执行

### 绝对禁止
- 交互式命令（文本编辑器、交互式安装向导）
- 绕过红线的任何手段

## 编码原则

### 简洁优先
- 只实现被要求的功能，不写投机性代码
- 单次使用的代码不做抽象；不要「未来可能用到」的灵活性
- 写了 200 行但 50 行能解决 → 重写

### 外科手术式修改
- 只改必须改的地方；不「顺手优化」无关代码
- 保持现有代码风格，即使你会用不同写法
- 改动产生的孤儿代码（无用 import/变量/函数）→ 主动清除

### 目标驱动执行
将任务转化为可验证的目标：
- 「修复 bug」→ 「写一个能复现它的测试，然后让它通过」
- 「重构 X」→ 「确保重构前后行为一致」

多步骤任务先列计划，每步附验证方式。

## 通用工程纪律

- 改完主动跑验证，不要只改不验
- 不要为了让代码跑起来注释掉报错或加绕过标记，找根本原因
- 密钥、token、密码不进代码、不进 commit、不进日志
- 大改动前先在 Plan Mode 出方案，确认后再动手
- 只有在有实质性改动的情况下才编辑文本. 改注释, 改空白符号等无实质性改动不要进行. 保持改动后的diff简洁明了, 适合人类review

## 错误处理

- **调用工具失败**：分析原因 → 尝试替代方案（Glob 失败 → 试 Grep）
- **连续失败 3 次**：向用户说明情况，不再重试
- **构建/测试失败**：增量修复，一次处理一个错误，每次修复后验证
- **不要静默吞错误**：异常退出码、空结果、意外行为都要提到

## 工作流

### 普通功能
明确需求 → 直接编码 → 验证

### 复杂功能 / 架构变更
出方案（Plan Mode / /grill-me）→ 用户确认 → 分阶段实现 → 全面验证 → 提交

## 嵌入式开发原则

- RAM很宝贵，尽量优化内存使用。需要大块RAM时(>256 Bytes)，显式地提醒用户。
- 使用RAM首选在堆上申请(malloc)，其次选栈上局部变量，全局数组是最差选择。
- 优化算法速度：用右移X代替除以2^X，不影响逻辑和损失性能的情况下，尽量避免除法/64位整型/浮点数/循环里的if。
